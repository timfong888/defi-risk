# Engineering Design & Plan

Consolidated engineering design for the Neutral DeFi Risk Intelligence
Aggregator. Companion docs: [poc-baseline.md](poc-baseline.md) (UI/IA),
[community-contribution-model.md](community-contribution-model.md)
(contribution mechanics), [CHARTER.md](../CHARTER.md) (binding constraints).

**Status: DRAFT — under design review** (see the Engineering Design Review
epic in Linear; this header updates to APPROVED with a version tag when
review completes).

## 1. Goals and constraints

Build the EF RFP's neutral aggregation layer: top-20 Ethereum DeFi protocols
× a 14-provider risk-feed registry, every cell assessed and labeled, feed
assessments verbatim. Binding constraints that shape the design:

- **No composite scoring** — enforced structurally: no schema field can hold
  a synthesized value (CHARTER.md).
- **Neutrality** — no editorial discretion in listing (mechanical inclusion
  criteria) or display (verbatim only, provenance-tagged).
- **Public good** — AGPL-3.0; the data layer must be independently useful
  (forkable, machine-readable, documented).
- **Thin stewardship budget** — $15k grant; every component is chosen to
  minimize standing maintenance (static site, serverless, PR-gated data).

## 2. Architecture

Static-first Next.js (App Router) on Vercel. No backend server, no database.

```
GitHub repo (source of truth)
├── data/*.json            hand-curated, provenance-tagged
├── data/synced/*.json     generated by scripts/sync-*.mjs (PR-gated)
├── scripts/               extraction scripts (re-runnable by anyone)
└── app/, components/      Next.js renderer
        │  build + ISR (revalidate: 3600)
        ▼
Vercel static pages ──► live metrics fetched server-side at revalidation
                        (DefiLlama TVL / 24h volume)
```

- **Writes** happen only via PRs to the repo (human or bot-opened).
- **Reads** are static pages; live metrics re-fetch at most hourly via ISR.
- **Failure mode honesty**: any fetch failure renders an explicit
  pending/stale badge — never a blank cell, never silent staleness.

## 3. Data model

| File | Contents | Written by |
|---|---|---|
| `data/protocols.json` | 20 seed protocols: family, versions, category, DefiLlama slug, metric kind (tvl / volume24h) | curated; new entries via mechanical inclusion rule |
| `data/feeds.json` | Feed registry: focus, type (Rating/Dashboard/Monitoring/Research), accessibility class + verified flag | curated; verification updates from SAT-302 |
| `data/coverage.json` | Manual matrix entries: status, provenance, note. Unlisted pairs default to `not-yet-covered` / `assessment-pending` in `lib/data.ts` — every cell resolves to a labeled state | curated |
| `data/synced/<feed>.json` | Generated verbatim assessments: status, verbatim text, source URL, updated date | sync scripts only (never hand-edited) |
| `data/details.json` | Per-protocol governance facts, audits, incidents — each with provenance + source | curated → progressively onchain-verified |
| `data/responses.json` (planned, M2) | Protocol right-of-response entries | protocol teams via PR |

**Merge precedence** (in `lib/data.ts`): synced (`provider-published`) >
manual (`manual-unverified`) > default (`assessment-pending`). Data quality
self-upgrades as feeds come online; no one deletes manual curation by hand.

**Provenance ladder** (rendered in UI): `onchain-verifiable` > `public-docs`
> `provider-published` > `self-reported` > `manual-unverified` >
`assessment-pending`.

## 4. Ingestion pipelines

### 4.1 Live metrics (built)
`lib/metrics.ts` fetches `api.llama.fi/tvl/{slug}` (TVL) or
`api.llama.fi/summary/aggregators/{slug}` (24h volume for the three swap
aggregators) server-side with `revalidate: 3600`. Null → explicit
"metric pending" badge.

### 4.2 Feed sync (pattern established; 1 of 14 live)
`scripts/sync-defiscan.mjs` is the template every feed follows:
fetch provider source → parse → normalize to the synced-entry schema →
write `data/synced/<feed>.json`. A nightly GitHub Action
(`.github/workflows/sync-feeds.yml`) re-runs scripts and opens a reviewable
PR on diff — no silent data mutation. Per-feed approach is driven by the
verified accessibility class (SAT-302):

| Class | Method | Provenance |
|---|---|---|
| public-api | direct fetch in sync script | provider-published |
| published-scrapeable | parser over published pages; breakage fails CI visibly | provider-published |
| gated-manual | curator checklist + reviewer sign-off; keys (if granted, SAT-304) live in Actions secrets | manual |

### 4.3 Governance extraction (planned, M1)
`scripts/extract-governance.ts` (viem): Safe `getOwners()`/`getThreshold()`,
timelock `getMinDelay()`, proxy admins, pause guardians — current-state
facts read directly from mainnet RPC, no indexer required. Event history
(proposals, parameter changes) later via The Graph subgraphs. Output merges
into `details.json` with `onchain-verifiable` provenance + contract address
+ method, so any reader can re-derive every fact.

## 5. CI and automation

- **Schema validation on every PR** (SAT-301): zod schemas shared between
  app and CI; malformed corrections fail visibly. This is the gatekeeper
  that makes mechanical listing and community correction safe.
- **Nightly feed sync** (built): PR-gated, reviewable diffs.
- **Staleness bot** (M2): opens issues on claimed protocol pages with data
  older than 90 days.
- **Threshold check** (M1): new-protocol PRs validated against the published
  inclusion criteria (DefiLlama slug resolves, TVL/volume above threshold).

## 6. Deployment and operations

- Vercel: `vercel deploy --prod` from main; preview deployments per PR.
- Rollback = redeploy previous build (static output, no migrations ever).
- Secrets: only in GitHub Actions secrets (feed API keys if granted) and
  Vercel env. Nothing secret in the repo; the app needs no keys at all today.
- Monitoring: Vercel analytics + the app's own honesty mechanics (stale
  badges surface data problems to every visitor — public accountability is
  the monitor).

## 7. Failure modes

| Failure | Behavior | Mitigation |
|---|---|---|
| DefiLlama down/changed | "metric pending" badge; page still serves | hourly retry via ISR; slug checks in CI |
| Provider repo/API schema change | sync script errors; nightly Action fails | Action failure notifies; manual curation fallback; provenance downgrades honestly |
| Spam/bad-faith data PRs | CI schema + provenance requirements reject | mechanical criteria; maintainer is liveness only |
| Provider rescinds data permission | remove verbatim, keep coverage status + link-out with explanatory note | per-provider agreements (SAT-307 pattern) |
| Steward goes inactive | data layer forkable (AGPL); site rebuildable by anyone | named-steward requirement + low-maintenance design |

## 9. Open questions

1. Scraper hosting for `published-scrapeable` feeds: in-repo Actions (free,
   public) vs. provider-negotiated exports (better; SAT-304 outcome decides).
2. RPC provider for governance extraction: public endpoints suffice for
   nightly current-state reads; confirm rate limits at 20 protocols.
3. Historical TVL sparklines: deferred — DefiLlama per-protocol history is
   heavyweight; revisit post-M1 if design review wants the density.
4. `.eth` mirror (eth.limo + IPFS static export): cheap because the site is
   static; decide post-M1 whether the differentiation is worth the extra
   deploy artifact.
